# Architecture: Handmade Studio

## 1. Краткий анализ product-brief.md

Handmade Studio — это студия кастомных проектов, а не обычный handmade-магазин. Основной фокус проекта: изделия из дерева, металла, гипса и комбинированных материалов для частных и коммерческих пространств.

Ключевые направления:

- изделия из дерева;
- интерьерный и экстерьерный декор;
- мебель и предметы для дома;
- таблички с названиями улиц и адресные таблички;
- оформление домов, бань, террас;
- оформление ресторанов, кафе и тематических пространств;
- индивидуальные проекты под задачу клиента.

Сайт должен показывать не только готовые изделия, но и способность студии проектировать, комбинировать материалы, адаптироваться под пространство и создавать предметы под конкретный объект. Основной язык запуска — русский, но архитектура с самого начала должна поддерживать RU / LV / EN.

## 2. Концепция сайта

Сайт Handmade Studio должен быть проектной витриной кастомной мастерской: посетитель видит реальные работы, понимает материалы и направления, быстро находит подходящий формат заказа и может связаться со студией.

Главная идея сайта: "создаём кастомные изделия и оформление пространств из дерева, металла, гипса и комбинированных материалов".

Основные задачи:

- показать уровень ремесла и проектного мышления;
- разделить направления работ по понятным категориям;
- объяснить процесс кастомного заказа;
- показать портфолио реальных объектов;
- привести пользователя к заявке через форму, WhatsApp, Telegram или Instagram;
- сразу заложить мультиязычность RU / LV / EN.

Целевая аудитория:

- владельцы домов, бань, террас и участков;
- владельцы ресторанов, кафе, гостевых домов и тематических пространств;
- дизайнеры интерьера и архитекторы;
- частные клиенты, которым нужны нестандартные изделия;
- коммерческие клиенты, которым нужно оформление пространства или вывесочные элементы.

## 5. Предлагаемая структура папок проекта

Рекомендуемая структура:

- `src/`
  - `app/` или `pages/` — маршруты страниц;
  - `components/` — базовые и UI-компоненты;
  - `sections/` — крупные секции страниц;
  - `content/` — тексты страниц, данные контента и переводы;
  - `data/` — услуги, проекты, материалы, отзывы, ссылки;
  - `i18n/` — языковая конфигурация RU / LV / EN, словари интерфейса и общие переводы;
  - `lib/` — вспомогательная логика;
  - `styles/` — глобальные стили и дизайн-токены;
  - `types/` — типы данных, если используется TypeScript.

Все тексты сайта должны храниться только в `i18n/` или `content/` слоях. Компоненты должны получать текст как данные или ключи переводов, а не содержать пользовательские строки напрямую.

- `public/`
  - `images/`
    - `hero/`
    - `projects/`
    - `materials/`
    - `services/`
    - `studio/`
    - `process/`
  - `icons/`
  - `favicon/`

- `docs/`
  - проектные заметки;
  - контентные требования;
  - инструкции для следующих агентов.

## 6. Хранение контента

Контент нужно отделить от компонентов, чтобы сайт можно было быстро переводить, пополнять и масштабировать.

### Тексты

Тексты хранить по языкам:

- `ru` — основной язык первого запуска;
- `lv` — подготовленный переводной слой;
- `en` — подготовленный переводной слой.

Для каждой страницы должны быть отдельные текстовые блоки:

- главная;
- о студии;
- услуги;
- портфолио;
- проект;
- контакты;
- SEO title и description.

### Услуги

Каждая услуга должна храниться как отдельная сущность:

- id;
- slug;
- название на RU / LV / EN;
- краткое описание на RU / LV / EN;
- подробное описание на RU / LV / EN;
- материалы;
- тип объекта;
- главное изображение;
- связанные проекты;
- CTA-текст;
- SEO-данные.

### Проекты

Каждый проект должен храниться как отдельная сущность:

- id;
- slug;
- название на RU / LV / EN;
- категория;
- материалы;
- тип объекта;
- локация, если можно показывать;
- задача клиента;
- решение;
- результат;
- главное фото;
- галерея;
- порядок отображения;
- SEO-данные.

### Материалы

Материалы лучше выделить отдельно:

- дерево;
- металл;
- гипс;
- комбинированные материалы.

Для каждого материала хранить:

- название на RU / LV / EN;
- описание;
- преимущества;
- где используется;
- связанные услуги;
- связанные проекты.

### Фото

Для каждого изображения хранить:

- путь к файлу;
- alt-текст на RU / LV / EN;
- принадлежность к проекту, услуге или материалу;
- тип изображения: hero, preview, gallery, process, detail;
- порядок отображения.

## 7. Подключение галереи

Галерея должна быть не просто набором фотографий, а инструментом просмотра проектных кейсов.

Рекомендуемая логика:

- основная галерея строится из проектов;
- каждый проект имеет главное фото и набор дополнительных изображений;
- фильтры работают по материалу, типу объекта и направлению;
- карточка проекта ведёт на детальную страницу;
- внутри проекта есть галерея деталей, процесса и финального результата;
- изображения загружаются оптимизированно;
- alt-тексты берутся из языкового слоя;
- порядок проектов задаётся вручную через поле sort/order.

Фильтры первого этапа:

- Все;
- Дерево;
- Металл;
- Гипс;
- Комбинированные материалы;
- Мебель;
- Декор;
- Таблички;
- Дома / бани / террасы;
- Рестораны / тематические пространства.

## 8. Контактная форма

Контактная форма должна быть ориентирована на проектную заявку, а не на покупку готового товара.

Рекомендуемые поля:

- имя;
- телефон или email;
- предпочтительный способ связи;
- тип проекта;
- материал, если клиент уже знает;
- краткое описание задачи;
- размеры / объект / место установки;
- сроки;
- согласие с политикой конфиденциальности.

Типы проекта в форме:

- изделие из дерева;
- мебель;
- декор;
- табличка;
- оформление дома / бани / террасы;
- оформление ресторана / коммерческого пространства;
- другое.

Поведение формы:

- обязательная валидация имени и контакта;
- понятные ошибки;
- состояние отправки;
- сообщение об успешной отправке;
- защита от спама;
- отправка заявки на email, в CRM или через serverless endpoint;
- возможность в будущем добавить прикрепление референсов.

## 9. WhatsApp / Telegram / Instagram

Контактные каналы нужно хранить в единой конфигурации, чтобы использовать их в Header, Footer, ContactSection и CTA-блоках.

Роли каналов:

- WhatsApp — быстрый запрос и первичная консультация;
- Telegram — обсуждение деталей, фото, размеров и референсов;
- Instagram — портфолио, доверие, процесс, свежие работы.

Рекомендуемые места размещения:

- верхняя навигация;
- футер;
- контактная страница;
- карточки услуг;
- страницы проектов;
- финальный CTA на главной;
- плавающая кнопка связи, если она не мешает просмотру портфолио.

Для WhatsApp и Telegram желательно подготовить предзаполненный текст: "Здравствуйте, хочу обсудить кастомный проект для Handmade Studio".

## 10. Подготовка под 3 языка: RU / LV / EN

Первый запуск делается на русском языке, но структура проекта сразу должна быть готова к латышскому и английскому.

Мультиязычность является обязательной частью архитектуры: все тексты, данные и компоненты должны поддерживать RU / LV / EN. Нельзя хранить пользовательский текст напрямую в компонентах; компоненты должны брать строки из `i18n` или `content` слоя.

Рекомендуемый подход:

- маршруты с языковым префиксом: `/ru`, `/lv`, `/en`;
- RU как язык по умолчанию на первом этапе;
- LanguageSwitcher в Header;
- все тексты вынести из компонентов;
- все услуги, проекты, материалы и SEO-поля хранить с переводами;
- alt-тексты изображений хранить на RU / LV / EN;
- не смешивать языки внутри одного текстового блока;
- предусмотреть fallback на RU, если LV или EN ещё не заполнены;
- URL-slug можно хранить отдельно для каждого языка или использовать единый стабильный slug.

Контентные сущности, которые должны быть мультиязычными:

- страницы;
- услуги;
- проекты;
- материалы;
- фильтры галереи;
- формы и ошибки валидации;
- SEO title / description;
- alt-тексты изображений;
- CTA-кнопки.
